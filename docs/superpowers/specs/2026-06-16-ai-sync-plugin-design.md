## 1. Цель

Отдельный CEP-плагин для Premiere Pro, который автоматически **синхронизирует клипы**
многокамерного/многодорожечного материала по **общему звуку** (кросс-корреляция
аудио-огибающих), с кадровой точностью (±1 кадр), превью офсетов перед применением
и безопасным откатом через checkpoint-секвенцию.

Плагин **самостоятельный**, не часть «ИИ: монтаж». Переиспользует проверенные паттерны
родителя (ffmpeg-препроцессинг, host-bridge с cold-start retry, CDP-отладка), но имеет
свой ExtensionBundleId, меню, порт отладки и тонкий host.

## 2. Результат L0-спайка (гейт пройден — ЗЕЛЁНЫЙ)

Перед разработкой проверена ключевая гипотеза хендофа §9: достижима ли кадровая
точность кросс-корреляцией огибающих. Спайк (`spike/l0-crosscorr.mjs`) на **реальном
клипе живого проекта** (25fps, 1 кадр = 40 мс):

| Сценарий | Худшая ошибка | Вывод |
|---|---|---|
| Чистый звук (известный τ) | **0.00 кадра** | RMS-огибающая 5мс + парабол. интерполяция → точно |
| Аддитивный шум до 100% RMS | **0.00 кадра** (corr падает до ~0.4) | устойчиво, деградирует только confidence |
| «Другой микрофон»: эхо 60мс + EQ | **0.00 кадра** (corr 0.95) | искажение формы огибающей не ломает пик |
| Реверберация (3 отражения) | **0.00 кадра** (corr 0.94) | главный риск хендофа снят |

**Вывод:** RMS-огибающая с окном **5 мс** + нормализованная кросс-корреляция +
параболическая интерполяция пика **достаточны для ±1 кадра**. PCM-уровневая корреляция
в продукте **не нужна**. Алгоритм спайка переносится в `sync-core.js` как основа.

## 3. Объём MVP (по ответам пользователя)

- **Материал:** много клипов на одной дорожке (перезапуски камер). Подтверждено живым
  проектом (11–17 клипов на дорожку). → Синхронизация работает **по-клипно**, а не
  по-дорожечно. Это полная модель (уровень L3 хендофа), а не упрощённый L2.
- **Превью:** текст **+ визуальный вейвформ** (canvas огибающих до/после выравнивания).
- **Ripple:** после выравнивания — **ripple-закрытие пауз** (убрать дыры/перекрытия,
  образовавшиеся при сдвиге клипов на дорожке).
- **Разные fps источников:** поддержка **с самого начала**. Камеры/рекордеры могут
  снимать на разных кадровых частотах (24/25/30/50/60). Работаем в **домене времени**
  (секунды → тики), сдвиг fps-независим; fps секвенции — только для отображения «±1 кадр».
- **Дрейф часов записи:** поддержка **с самого начала** (см. §5.7). Реальный кейс для
  подкастов: длинные записи (1–3 ч) на разных рекордерах расходятся 5–50 ppm
  (до ~180 мс = 4.5 кадра за час). Детекция — всегда; коррекция rate-stretch —
  по порогу. Это ключевое отличие профи-инструментов (PluralEyes/Syncaila) от
  встроенной синхронизации Premiere.

### 4.2. Модули (`client/shared/`, IIFE ES5, экспорт на `window`)

| Модуль | Происхождение | Роль |
|---|---|---|
| `audio-envelope.js` | адаптация `audio-preprocess.js` родителя | ffmpeg → моно-PCM → **RMS-огибающая @5мс**; поиск ffmpeg, no-op без Node |

| `sync-graph.js` | **новый** | выбор опорной (anchor) дорожки; по-клипное разрешение офсетов; confidence-гейтинг; **флаг дрейфа** по клипам |

| `sync-waveform.js` | **новый** | canvas-рендер огибающих до/после выравнивания |
| `bridge-premiere.js` | адаптация родителя | загрузка host JSX + cold-start retry + evalJson |
| `panel.js` + `index.html` + `styles.css` | новый / адаптация | UI панели: кнопка анализа, таблица превью, вейвформ, Apply, Revert |

### 4.3. Тонкий host (`host/premiere-sync.jsx`)

Извлекается из родительского `premiere.jsx`, **только нужные функции**, каждая
обёрнута декоратором `_wrap` (структурированный `{_hostError}` при исключении):

- `getTimelineSnapshot()` — дорожки, клипы, mediaPath, in/out, start/end, ticks, fps.
- `getClipMediaPath(nodeId)` — путь к медиа клипа.
- `moveClip(nodeId, deltaTicks)` — сдвиг клипа на субкадровое смещение (тики).
  Связанные A/V двигаются вместе (host обрабатывает linkage).
- `rippleCloseGaps(trackIndex, trackType)` — после сдвигов закрыть дыры/перекрытия на
  дорожке (ripple-уплотнение в порядке клипов).
- `setClipSpeed(nodeId, ratio)` — растяжка клипа по скорости для коррекции дрейфа
  (`ratio = 1 + slope`). Самая сложная операция; если нативно нестабильно — фолбэк
  на рендер скорректированного аудио через ffmpeg `atempo` (см. §5.7, R5).
- `backupActiveSequence()` — клон активной секвенции как checkpoint перед Apply.
- `activateSequenceById(seqId)` — Revert на checkpoint.

Host держим **тонким** намеренно — ограничиваем долг будущей миграции на UXP (хендоф §8).

## 5. Модель по-клипной синхронизации (ядро)

1. **Снимок:** `getTimelineSnapshot()` → все клипы с mediaPath, in/out, start/end в
   секундах и тиках, fps секвенции.
2. **Опора (anchor):** дорожка с максимальным аудио-покрытием/энергией. Её клипы = офсет 0.
3. **Непрерывная огибающая опоры:** конкатенация огибающих клипов опоры по sequence-time
   (с учётом ремапа media→sequence).
4. **По каждому неопорному клипу:** извлечь огибающую → кросс-корреляция с огибающей
   опоры в окне поиска → офсет τ + confidence (значение пика corr).
5. **Confidence-гейт:** ниже порога → клип помечается «не синхронизировано, проверьте
   вручную»; **молча не двигаем**.
6. **Превью:** карточка (текст + вейвформ) → пользователь проверяет → **Apply**:
   `backupActiveSequence` (checkpoint) → `moveClip` каждого клипа на его τ (тики,
   субкадрово) → опц. `setClipSpeed` для дрейфующих клипов → `rippleCloseGaps` для
   уплотнения дорожек. **Revert** → `activateSequenceById`.

### 5.7. Разные fps и дрейф часов

- **Fps-агностичность.** Вся математика синхронизации — в секундах; сдвиг применяется в
  тиках (универсальная единица Premiere, 254016000000 тик/сек). fps каждого источника
  берётся из снимка, но на сам офсет не влияет. Сдвиг fps между источниками работает
  «бесплатно» при работе в домене времени.
- **Детекция дрейфа (всегда, дёшево).** Для каждого клипа корреляция считается в **двух
  окнах** — у начала и у конца клипа. Если офсет начала τ₀ ≠ офсет конца τ₁, есть дрейф:
  `slope = (τ₀ − τ₁) / clipDurationSec` (знак выбран так, чтобы `ratio = 1 + slope`
  компенсировал дрейф). Флаг ставится, если |τ₁ − τ₀| > 1 кадра.
- **Коррекция (по порогу).** Только если дрейф превышает порог: `setClipSpeed(nodeId,
  1 + slope)` растягивает клип так, чтобы концы сошлись. Применяется опционально, с
  пометкой в превью. Ниже порога — дрейф игнорируется (один офсет на клип).
- **Порог по умолчанию:** 1 кадр на длину клипа (настраиваемо). Подкастовые часы
  записи (1–3 ч) почти всегда превышают порог → коррекция реально нужна.

## 6. Порядок сборки (walking skeleton)

Многоклиповая синхронизация — конечный результат; скелет — лишь *как* строим.

- **P0 — водопровод:** scaffold + тонкий host + bridge. Доказать в живом Premiere
  сквозной проход snapshot → огибающая → xcorr → move → backup на тривиальном случае.
- **P1 — опора + по-клипно:** полный матчинг всех клипов (fps-агностично) против
  огибающей опоры, confidence-гейтинг, окно поиска как параметр.
- **P2 — ripple + дрейф:** `rippleCloseGaps` после сдвигов; детекция дрейфа (две корреляции)
  и `setClipSpeed`-коррекция по порогу с фолбэком на ffmpeg `atempo`.
- **P3 — превью:** canvas-вейвформ (до/после), пакетный Apply, Revert, краевые случаи.

## 7. Тестирование

- **Unit** (`npm test`, vm-loader как у родителя): регрессии `sync-core` —
  - синтетический известный сдвиг → ±1 кадр (seed-тест из спайка),
  - шум / реверберация / эхо (деградирует только confidence, не точность),
  - тишина → нет ложного матча (низкий corr),
  - идентичное аудио → detect (высокий corr, τ≈0),
  - дрейф: растянутая копия с известным наклоном → детектор восстанавливает slope;
    короткий клип без дрейфа → slope≈0 (нет ложной коррекции).
- **Live** через `cep-debug.mjs` (порт 8099) против открытого проекта пользователя:
  snapshot, по-клипный матчинг, Apply/Revert.

## 8. Риски

- **R1 — клип не перекрывается с опорой.** При перезапусках камеры клип в текущей
  позиции может не иметь общего звука с опорой → нужен широкий глобальный поиск.
  *Митигация:* окно поиска как параметр; слабые матчи гейтим, не двигаем вслепую.
- **R2 — несколько опорных кандидатов / разрывы в опоре.** Опорная дорожка тоже может
  иметь дыры. *Митигация:* опора выбирается по покрытию; разрывы — отдельные сегменты
  огибающей, поиск идёт по ближайшему сегменту.
- **R3 — ripple-закрытие пауз.** Сдвиг клипов оставляет дыры/перекрытия; ripple-уплотнение
  на одной дорожке не должно ломать выравнивание (абсолютные позиции уже синхронизированы).
  *Митигация:* ripple применяется **после** всех сдвигов, по-дорожечно, с пересчётом
  превью; checkpoint-секвенция даёт откат.
- **R4 — миграция на UXP.** CEP уходит. *Митигация:* тонкий host, изоляция ExtendScript
  за bridge (хендоф §8).
- **R5 — коррекция дрейфа в host.** Нативный `setClipSpeed` в ExtendScript нестабилен и
  меняет длительность клипа. *Митигация:* фолбэк — рендер скорректированного аудио через
  ffmpeg `atempo=1+slope` и подмена/импорт; либо ограничить P2 детекцией+предупреждением,
  а rate-stretch вынести за гейт-проверку стабильности нативного API.

## 9. Не входит в объём (YAGNI)

- ИИ/LLM, агентный цикл, чат — это родительский плагин.
- Транскрипция, монтаж по тексту, маркеры, экспорт.
- Multicam-перепривязка через нативный Premiere Multicam Source Sequence (только
  сдвиг позиций клипов + опц. rate-stretch для дрейфа).
- Нелинейный дрейф (скачки тактовой частоты в середине клипа) — компенсируем только
  **линейный** дрейф (один наклон на клип). Нелинейный — будущая фаза.
